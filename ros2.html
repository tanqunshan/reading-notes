Robot Operating System 2 · Architecture & Applications

技术学习笔记 · 整理日期：2026年07月

开源机器人框架 DDS 通信 Humble / Jazzy LTS

一、ROS 2 是什么？

ROS 2（Robot Operating System 2）是新一代开源机器人软件框架，由 Open Robotics 主导开发。它并非传统意义上的「操作系统」，而是一套中间件 + 工具链 + 生态包，帮助开发者快速构建模块化、分布式的机器人系统。

ROS 1 曾广泛用于科研与原型开发，但存在Master 单点故障、实时性不足、安全性弱、难以产品化等局限。ROS 2 从零设计，以工业级 DDS（Data Distribution Service）为通信骨干，实现去中心化、可配置 QoS、跨平台与产品级部署能力。

ROS 2 的设计目标：减少重复造轮子，让开发者专注业务逻辑，而非底层通信与集成。—— Open Robotics

二、分层架构（自上而下）

ROS 2 采用清晰的分层设计，上层应用与底层通信实现解耦，可通过 RMW 接口切换不同 DDS 实现（如 Fast DDS、Cyclone DDS）。

应用层Package · Node · Launch · Parameter
RCL 客户端库rclcpp (C++) · rclpy (Python) · 统一 ROS API
RMW 中间件抽象ROS Middleware Interface · 可切换 DDS 实现
DDS 实现层Fast DDS · Cyclone DDS · Connext 等
操作系统 / 硬件Linux · Windows · macOS · RTOS · MCU

【与 ROS 1 的关键差异】

维度ROS 1ROS 2
通信架构Master 中心化（roscore）DDS 去中心化，无单点故障
实时性较弱QoS 可配置，支持实时控制
安全性基本无DDS-Security（SROS2）加密与认证
跨平台主要 LinuxLinux / Windows / macOS / RTOS / MCU
多机器人支持有限原生支持多机协同
产品化偏科研原型面向工业与消费级部署

三、核心概念

【ROS Graph · 计算图】

ROS 2 系统由节点及其连接构成一张计算图（Graph）：每个节点负责单一、模块化的功能（如相机驱动、路径规划），通过 Topic、Service、Action、Parameter 与其他节点交换数据。可用 ros2 node list、ros2 topic list 等 CLI 工具 inspect 整张图。

📡

Topic 话题

发布/订阅 · 异步单向 · 适合传感器流、状态广播

🔄

Service 服务

请求/响应 · 同步 RPC · 适合一次性查询或配置

🎯

Action 动作

目标+反馈+结果 · 长时任务 · 可取消 · 适合导航、抓取

⚙

Parameter 参数

节点运行时配置 · 支持动态修改 · 无需重编译

【Node 节点】

节点是 ROS 2 的基本执行单元，遵循「一个节点一个职责」。一个可执行文件可包含多个节点。节点通过命名空间（Namespace）与重映射（Remapping）组织，避免 Topic 名冲突。

【Package 功能包】

功能包是代码的组织单元，包含节点源码、消息定义、Launch 文件、配置文件等。工作空间（Workspace）用 colcon 构建，生成 build/、install/、log/ 三个目录。

【Launch 启动】

ROS 2 使用 Python 编写的 .launch.py 文件统一启动多节点、配置参数与重映射，替代 ROS 1 的 XML Launch，更灵活可编程。

【DDS · 通信基石】

DDS 采用以数据为中心的发布/订阅（DCPS）模型：所有节点在全局 DataSpace 中发布和订阅，通过 Domain ID 隔离不同系统（只有同一 Domain 内的节点才能通信）。相比 ROS 1 的 Broker/Master 模型，DDS 无中心瓶颈，节点故障不影响其他节点发现与通信。

【QoS · 服务质量策略】

QoS 是 DDS 的核心特性，发布者与订阅者之间形成「合约」，控制数据传输行为。常用策略包括：

  • Reliability：RELIABLE（可靠传输） vs BEST_EFFORT（尽力传输，低延迟）
  • Durability：是否保留历史数据供后加入的订阅者读取
  • History：KEEP_LAST / KEEP_ALL，控制缓存深度
  • Deadline / Lifespan：数据时效性约束

传感器数据常用 BEST_EFFORT + KEEP_LAST；控制指令常用 RELIABLE，以匹配不同业务场景。

六、设计原则与最佳实践

  • 单一职责：每个节点只做一件事，便于测试、替换与复用
  • 接口标准化：通过标准消息类型（sensor_msgs、geometry_msgs 等）降低集成成本
  • QoS 匹配：发布者与订阅者 QoS 须兼容，否则无法建立连接
  • TF 坐标变换：统一多传感器、多关节的空间关系（tf2 库）
  • 生命周期节点：Managed Node 支持有序启动/关闭，适合产品级系统
  • Composable Node：同进程多节点零拷贝通信，降低延迟
  • 仿真先行：Gazebo / Isaac Sim 验证后再上真机，降低风险

七、学习心得

理解 ROS 2，关键是抓住「分层 + 去中心化 + 可配置」三个词。DDS 替换了 ROS 1 的 Master，不仅消除了单点故障，还通过 QoS 让同一套框架既能传高频传感器流，也能传可靠的控制指令——这是走向工业落地的技术基础。

Topic / Service / Action 三种通信模式的选择，本质上是对数据语义的选择：流式数据用 Topic，一次性查询用 Service，长时任务用 Action。很多集成问题，根源是通信模式选错或 QoS 不匹配。

从应用场景看，ROS 2 已不局限于实验室：仓储 AMR、产线复合机器人、人形机器人二次开发、自动驾驶感知栈，都在以 ROS 2 为「软件基础设施」。Nav2 + MoveIt 2 + ros2_control 构成移动与操作机器人的标准三板斧；ROS-Industrial 则补齐了手眼标定、可达性分析等工业特有能力。

对于 RCS（机器人控制系统）类项目，ROS 2 的价值在于：它提供了一套可复用的模块化语言——节点如何拆分、消息如何定义、任务如何调度，都有社区共识。不必从零设计通信协议，而可专注在业务逻辑（调度、路径、设备适配）上。这也是为何越来越多物流与制造场景选择 ROS 2 作为上层集成框架的原因。
